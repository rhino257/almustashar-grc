# ADR-0010: المعمارية التطوّرية للمنصّة (أحادي معياري ← وكلاء ← بوابة B2B)

## الحالة
Accepted — 2026-07-11

## السياق

تطوّر نطاق «المستشار القانوني الذكي» من *backend لتطبيق واحد* إلى **منصّة**
يُتوقّع منها مستقبلاً أن:

1. تشغّل **أكثر من وكيل (multi-agent)** فوق منظومة RAG الحالية.
2. تقدّم **خدمة API للشركات (B2B)** تتصل بنموذجنا.
3. تضيف **خدمات ذكاء اصطناعي إضافية** مثل OCR وغيرها.

القيود الحاكمة ثابتة: On-Prem كامل على «سحبكم» (اليمن)، 1x NVIDIA H100 80GB،
فريق صغير، تزامن ~50، ومبدأ **«البرمجة على واجهة»** المُثبت في ARC-DESIGN-001.

السؤال المعماري: كيف نبني من أول مرة بحيث نستوعب هذا النمو دون إعادة هيكلة،
ودون أن ندفع كلفة تعقيد الخدمات المصغّرة قبل أوانها؟

## محرّكات القرار

- تجنّب إعادة الهيكلة المكلفة لاحقاً (الدقّة من أول مرة).
- ملاءمة فريق صغير وبنية GPU واحدة (كلفة تشغيلية منخفضة).
- عزل واضح للحدود بين النطاقات يسمح باستخراج خدمات لاحقاً برخص.
- فصل ما هو مقيّد بالـ GPU (الاستدلال) عمّا هو منطق تطبيق.
- حماية تطوّر التطبيق الداخلي من التزامات عقد الـ B2B العام.

## الخيارات المدروسة

1. **خدمات مصغّرة (Microservices) منذ البداية.**
2. **أحادي تقليدي (Monolith بسيط) دون حدود معيارية.**
3. **أحادي معياري (Modular Monolith) + طبقة استدلال مفصولة + حدود ports/adapters جاهزة للاستخراج.** ✅

## القرار

نعتمد الخيار (3): **معمارية تطوّرية على ثلاث مراحل**.

**المبدأ الحاكم:** نواة أحادية معيارية نظيفة (FastAPI) + **طبقة استدلال منفصلة**،
مع حدود ports/adapters. كل قدرة تحتاج GPU (النموذج، OCR، …) تدخل كـ
**خدمة استدلال مستقلة خلف عقد REST ثابت**، لا داخل النواة.

### المراحل

- **المرحلة 1 (الآن):** النواة الأحادية المعيارية + RAG كوحدة داخلية
  (`app/rag/`) + طبقة استدلال منفصلة (vLLM/Embeddings/Reranker). تُبنى بهيكل
  مستوحى من Netflix Dispatch (تنظيم حسب النطاق) وطبقات Clean/DDD.
- **المرحلة 2 (تعدّد الوكلاء):** إضافة `app/agents/` بنمط **supervisor** في
  LangGraph — وكيل موجِّه يوزّع على وكلاء متخصّصين، كلٌّ خلف واجهة. لا يُبدأ
  بتعدّد الوكلاء قبل اتّضاح حدود القدرات.
- **المرحلة 3 (منصّة B2B):** إضافة **API Gateway** أمام النواة يوفّر: مفاتيح API
  لكل شركة، rate limiting لكل مفتاح (token bucket / Redis)، تعدّد مستأجرين عبر
  **shared schema + RLS** (امتداد لنموذجنا بإضافة `tenant_id`)، وقياس استهلاك
  للفوترة. كل خدمة جديدة (OCR…) تُضاف كخدمة استدلال مُحاوية مستقلة خلف عقد REST.

### الهيكل المعتمد (إضافات على backend/)

​
app/
├── modules/            # نطاقات العمل
├── agents/             # مرحلة 2: وكلاء LangGraph (supervisor + subgraphs)
│   ├── supervisor.py
│   └── specialists/    # legal_advisor, researcher, summarizer ...
├── inference/          # محوّلات (adapters) لكل خدمة استدلال خلف ports
│   ├── ports.py        # LLMClient / EmbeddingClient / RerankerClient / OcrClient
│   └── adapters/       # vllm, ollama, embeddings, jina, ocr_service
├── platform/           # مرحلة 3: مفاتيح API، tenancy، rate limiting، usage metering
└── api/
├── v1/             # عقد التطبيق الداخلي (ARC-API-001)
└── public/         # مرحلة 3: عقد الـ API العام للشركات (منفصل ومُصدَّر)

**قاعدة ملزِمة:** يُفصل **عقد التطبيق الداخلي** (`api/v1`) عن **العقد العام للشركات**
(`api/public`) منذ اليوم الأول، حتى لا يتقيّد تطوّر التطبيق بوعود عقد B2B.

## النتائج (Consequences)

### إيجابية
- بداية بسيطة منخفضة الكلفة تناسب فريقاً صغيراً وبنية GPU واحدة.
- حدود ports/adapters تجعل استخراج أي خدمة (RAG، OCR) لاحقاً رخيصاً.
- فصل الاستدلال (GPU-bound) عن منطق التطبيق يسمح بتوسيعه مستقلاً.
- توافق مباشر مع مبدأ «البرمجة على واجهة» ومع RLS القائم.

### سلبية / مقايضات
- الأحادي المعياري يتطلّب انضباطاً في احترام حدود الوحدات (لا اختصارات عبر النطاقات).
- تأجيل الخدمات المصغّرة يعني أن أي استخراج مستقبلي يتطلّب عمل ترحيل مقصوداً.
- طبقة الـ Gateway وتعدّد المستأجرين تضيف تعقيداً يُفعَّل في المرحلة 3 فقط.

## المعايير المرجعية

- Netflix Dispatch وfastapi-best-practices (التنظيم حسب النطاق).
- Clean Architecture / DDD وModular Monolith Primer.
- LangGraph (أنماط supervisor/hierarchical لتعدّد الوكلاء).
- نمط AI Gateway (بوابة موحّدة: توجيه + مصادقة + rate limiting).
- نمط الخدمة الاستدلالية المُحاوية خلف عقد REST ثابت (لـ OCR والخدمات المماثلة).
